# Guia de configuração LDAP do Nexus com Active Directory

## 1. Objetivo

Este documento descreve como configurar o Sonatype Nexus Repository para autenticar usuários via LDAP usando o Active Directory do homelab JMSalles.

O foco é permitir que usuários do AD acessem o Nexus usando seus logins do domínio:

```text
jmsalles
adm-jeferson
```

O domínio utilizado será:

```text
jmsalles.homelab.br
```

O Nexus utilizado no ambiente responde por:

```text
https://nexus.jmsalles.homelab.br
```

## 2. Dados do ambiente

| Item                    | Valor                                                                          |
| ----------------------- | ------------------------------------------------------------------------------ |
| Nexus                   | `https://nexus.jmsalles.homelab.br`                                            |
| Domínio AD              | `jmsalles.homelab.br`                                                          |
| Domain Controller       | `winserver.jmsalles.homelab.br`                                                |
| IP do Domain Controller | `192.168.31.24`                                                                |
| Porta LDAP              | `389`                                                                          |
| Protocolo inicial       | `ldap://`                                                                      |
| Search Base DN          | `DC=jmsalles,DC=homelab,DC=br`                                                 |
| User Relative DN        | `OU=Homelab`                                                                   |
| Group Relative DN       | `OU=Groups,OU=Homelab`                                                         |
| Bind DN                 | `CN=svc-ldap-bind,OU=Service Accounts,OU=Homelab,DC=jmsalles,DC=homelab,DC=br` |
| Conta de bind           | `svc-ldap-bind`                                                                |
| Grupo admin do AD       | `GRP_HOMELAB_ADMINS`                                                           |
| Usuários esperados      | `jmsalles`, `adm-jeferson`                                                     |

## 3. Estratégia adotada

A integração será feita em três etapas:

```text
1. Habilitar o LDAP Realm no Nexus
2. Criar a conexão LDAP com o Active Directory
3. Criar o mapeamento do grupo AD para uma role administrativa no Nexus
```

O grupo usado para liberar acesso administrativo será:

```text
GRP_HOMELAB_ADMINS
```

A ideia é que usuários membros desse grupo tenham permissão administrativa no Nexus.

## 4. Observação importante sobre conta local admin

Antes de configurar LDAP, mantenha funcionando uma conta local administrativa do Nexus.

Exemplo:

```text
admin
```

ou uma conta local administrativa criada por você.

Não remova o login local antes de validar completamente o LDAP.

Motivo:

```text
Se a configuração LDAP ficar incorreta, a conta local permite recuperar o acesso administrativo ao Nexus.
```

## 5. Observação sobre Docker Registry

Como seu Nexus também é usado para Docker registry, não remova realms já utilizados pelo Docker.

Se existir o realm abaixo ativo, mantenha:

```text
Docker Bearer Token Realm
```

Esse realm costuma ser necessário para autenticação Docker em repositórios Docker hosted/proxy/group.

Neste guia, será adicionado o LDAP Realm sem remover os realms que já funcionam.

## 6. Validar conectividade com o AD

No servidor Docker HML, teste a porta LDAP:

```bash
nc -vz 192.168.31.24 389
```

Se o comando `nc` não existir:

```bash
sudo dnf install -y nmap-ncat
```

Teste também por nome:

```bash
nc -vz winserver.jmsalles.homelab.br 389
```

Validar DNS:

```bash
nslookup winserver.jmsalles.homelab.br
```

Se o nome não resolver corretamente a partir do servidor/container do Nexus, use o IP no campo Hostname:

```text
192.168.31.24
```

## 7. Testar LDAP antes de configurar o Nexus

No servidor Docker HML ou em outro Linux, instale o cliente LDAP:

```bash
sudo dnf install -y openldap-clients
```

Teste a conta de bind:

```bash
ldapwhoami -x \
  -H ldap://192.168.31.24:389 \
  -D "svc-ldap-bind@jmsalles.homelab.br" \
  -W
```

Resultado esperado:

```text
u:JMSALLES\svc-ldap-bind
```

Teste o usuário `jmsalles`:

```bash
ldapsearch -x \
  -H ldap://192.168.31.24:389 \
  -D "svc-ldap-bind@jmsalles.homelab.br" \
  -W \
  -b "OU=Homelab,DC=jmsalles,DC=homelab,DC=br" \
  "(&(objectClass=user)(sAMAccountName=jmsalles))" \
  distinguishedName sAMAccountName userPrincipalName memberOf
```

Teste o usuário `adm-jeferson`:

```bash
ldapsearch -x \
  -H ldap://192.168.31.24:389 \
  -D "svc-ldap-bind@jmsalles.homelab.br" \
  -W \
  -b "OU=Homelab,DC=jmsalles,DC=homelab,DC=br" \
  "(&(objectClass=user)(sAMAccountName=adm-jeferson))" \
  distinguishedName sAMAccountName userPrincipalName memberOf
```

O retorno deve mostrar o usuário e o grupo:

```text
CN=GRP_HOMELAB_ADMINS,OU=Groups,OU=Homelab,DC=jmsalles,DC=homelab,DC=br
```

## 10. Tela User and Group Settings

A segunda etapa configura como o Nexus encontra usuários e grupos no AD.

A documentação da Sonatype indica que o mapeamento de usuários define atributos como username, nome real e e-mail, e que o Password attribute normalmente deve ficar em branco quando a autenticação ocorre via bind no LDAP.

## 10.1 Configuration template

Se existir o campo:

```text
Configuration template
```

Selecione:

```text
Active Directory
```

Depois ajuste os campos manualmente conforme abaixo.

## 10.2 User Element Mapping

Campo:

```text
User relative DN
```

Valor:

```text
OU=Homelab
```

Campo:

```text
User subtree
```

Valor:

```text
Marcado
```

Campo:

```text
Object class
```

Valor:

```text
user
```

Campo:

```text
User filter
```

Valor recomendado para restringir aos administradores do homelab:

```text
(&(objectClass=user)(sAMAccountName=*)(memberOf=CN=GRP_HOMELAB_ADMINS,OU=Groups,OU=Homelab,DC=jmsalles,DC=homelab,DC=br))
```

Esse filtro limita a listagem/autenticação aos usuários membros do grupo:

```text
GRP_HOMELAB_ADMINS
```

Se quiser testar de forma mais aberta no primeiro momento, use temporariamente:

```text
(objectClass=user)
```

Depois volte para o filtro restritivo.

Campo:

```text
User ID attribute
```

Valor:

```text
sAMAccountName
```

Campo:

```text
Real name attribute
```

Valor:

```text
cn
```

Campo:

```text
Email attribute
```

Valor recomendado no seu homelab:

```text
userPrincipalName
```

Alternativa futura, se o atributo `mail` estiver preenchido no AD:

```text
mail
```

Campo:

```text
Password attribute
```

Valor:

```text
Deixar em branco
```

Motivo:

```text
O Nexus autenticará o usuário fazendo bind no LDAP.
Não é necessário mapear atributo de senha.
```

## 10.3 Group Element Mapping

A documentação da Sonatype descreve grupos LDAP como static ou dynamic. Para grupos estáticos, o grupo contém uma lista explícita de membros e o Group member format pode ser usado para extrair o usuário do atributo de membro.

Campo:

```text
Group type
```

Valor:

```text
Static Groups
```

Campo:

```text
Group relative DN
```

Valor:

```text
OU=Groups,OU=Homelab
```

Campo:

```text
Group subtree
```

Valor:

```text
Marcado
```

Campo:

```text
Group object class
```

Valor:

```text
group
```

Campo:

```text
Group ID attribute
```

Valor:

```text
cn
```

Campo:

```text
Group member attribute
```

Valor:

```text
member
```

Campo:

```text
Group member format
```

Valor:

```text
${dn}
```

Explicação:

```text
No Active Directory, o atributo member dos grupos normalmente armazena o Distinguished Name completo dos usuários.
Por isso, o formato ${dn} é o mais adequado para este cenário.
```

## 11. Habilitar o LDAP Realm

Depois da conexão LDAP criada, habilite o realm.

Caminho:

```text
Settings
Security
Realms
```

Na lista de realms disponíveis, localize:

```text
LDAP Realm
```

Mova para a lista de realms ativos.

A Sonatype orienta adicionar o LDAP Realm aos active realms e manter o Local Authenticating Realm ativo, deixando o LDAP abaixo dele.

Ordem recomendada:

```text
Local Authenticating Realm
Local Authorizing Realm
LDAP Realm
Docker Bearer Token Realm
```

Observação:

```text
A ordem pode variar conforme os realms existentes no seu Nexus.
Não remova realms que já estão sendo usados pelo Docker registry.
```

Clique em:

```text
Save
```

## 22. Troubleshooting

### 22.1 Verify connection falha

Teste a porta no host Docker HML:

```bash
nc -vz 192.168.31.24 389
```

Valide DNS:

```bash
nslookup winserver.jmsalles.homelab.br
```

Se o hostname falhar, use no Nexus:

```text
192.168.31.24
```

### 22.2 Erro de senha ou bind

Teste a conta de bind:

```bash
ldapwhoami -x \
  -H ldap://192.168.31.24:389 \
  -D "svc-ldap-bind@jmsalles.homelab.br" \
  -W
```

Se falhar, valide:

```text
Senha do svc-ldap-bind
Conta habilitada
Conta bloqueada
DN correto
```

### 22.3 Verify user mapping não encontra usuários

Teste a busca manual:

```bash
ldapsearch -x \
  -H ldap://192.168.31.24:389 \
  -D "svc-ldap-bind@jmsalles.homelab.br" \
  -W \
  -b "OU=Homelab,DC=jmsalles,DC=homelab,DC=br" \
  "(&(objectClass=user)(sAMAccountName=jmsalles))" \
  distinguishedName sAMAccountName memberOf
```

Se o usuário aparecer no `ldapsearch` mas não aparecer no Nexus, revise:

```text
User relative DN
User subtree
Object class
User filter
User ID attribute
```

### 22.4 Usuário existe, mas não passa no filtro

Se usar este filtro:

```text
(&(objectClass=user)(sAMAccountName=*)(memberOf=CN=GRP_HOMELAB_ADMINS,OU=Groups,OU=Homelab,DC=jmsalles,DC=homelab,DC=br))
```

o usuário precisa estar diretamente no grupo:

```text
GRP_HOMELAB_ADMINS
```

Valide:

```bash
ldapsearch -x \
  -H ldap://192.168.31.24:389 \
  -D "svc-ldap-bind@jmsalles.homelab.br" \
  -W \
  -b "OU=Homelab,DC=jmsalles,DC=homelab,DC=br" \
  "(sAMAccountName=jmsalles)" \
  memberOf
```

### 22.5 Login funciona, mas sem permissão

Valide se o LDAP Realm está ativo:

```text
Settings
Security
Realms
```

Valide se a role externa foi criada:

```text
Settings
Security
Roles
```

Valide se o grupo externo foi mapeado:

```text
GRP_HOMELAB_ADMINS -> nx-admin
```

### 22.6 Grupo não aparece no mapeamento externo

Valide o mapeamento de grupos na conexão LDAP.

Revise:

```text
Group type
Group relative DN
Group object class
Group ID attribute
Group member attribute
Group member format
```

Teste grupo via LDAP:

```bash
ldapsearch -x \
  -H ldap://192.168.31.24:389 \
  -D "svc-ldap-bind@jmsalles.homelab.br" \
  -W \
  -b "OU=Groups,OU=Homelab,DC=jmsalles,DC=homelab,DC=br" \
  "(cn=GRP_HOMELAB_ADMINS)" \
  cn distinguishedName member
```

### 22.7 Erro de e-mail vazio

Se o Nexus reclamar de e-mail ou não montar usuário corretamente, use:

```text
userPrincipalName
```

no campo:

```text
Email attribute
```

Se futuramente você preencher o atributo `mail` no AD, poderá trocar para:

```text
mail
```

### 22.8 Login local admin parou

O login local deve continuar funcionando se o Local Authenticating Realm permanecer ativo.

Valide:

```text
Settings
Security
Realms
```

Mantenha:

```text
Local Authenticating Realm
Local Authorizing Realm
```

ativos.

### 22.9 LDAP simples em porta 389

Neste momento, a configuração usa LDAP simples:

```text
ldap://192.168.31.24:389
```

Para homelab, isso facilita os testes.

Futuramente, o ideal é evoluir para LDAPS:

```text
ldaps://winserver.jmsalles.homelab.br:636
```

Para isso será necessário:

```text
Configurar certificado no Domain Controller
Instalar a CA raiz no container/host do Nexus
Alterar protocolo para ldaps://
Alterar porta para 636
Validar cadeia de confiança
```
