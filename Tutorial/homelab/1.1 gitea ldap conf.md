# Guia de configuração de autenticação LDAP do Gitea com Active Directory

## 1. Objetivo

Este documento descreve como configurar o Gitea para autenticar usuários via LDAP usando o Active Directory do homelab JMSalles.

O foco deste guia é a configuração da tela:

```text
Configurações de Administrador
Identidade e Acesso
Fontes de Autenticação
Adicionar Fonte de Autenticação
```

O Gitea usará o Active Directory do domínio:

```text
jmsalles.homelab.br
```

## 2. Dados do ambiente

| Item                    | Valor                                                                          |
| ----------------------- | ------------------------------------------------------------------------------ |
| Gitea                   | `git.jmsalles.homelab.br`                                                      |
| Domínio AD              | `jmsalles.homelab.br`                                                          |
| Domain Controller       | `winserver.jmsalles.homelab.br`                                                |
| IP do Domain Controller | `192.168.31.24`                                                                |
| Porta LDAP              | `389`                                                                          |
| Protocolo inicial       | `Unencrypted`                                                                  |
| Base DN                 | `DC=jmsalles,DC=homelab,DC=br`                                                 |
| Base de usuários        | `OU=Homelab,DC=jmsalles,DC=homelab,DC=br`                                      |
| Base de grupos          | `OU=Groups,OU=Homelab,DC=jmsalles,DC=homelab,DC=br`                            |
| Usuário de bind         | `svc-ldap-bind`                                                                |
| Bind DN                 | `CN=svc-ldap-bind,OU=Service Accounts,OU=Homelab,DC=jmsalles,DC=homelab,DC=br` |
| Grupo admin AD          | `GRP_HOMELAB_ADMINS`                                                           |

## 3. Estratégia adotada

Neste primeiro momento, a configuração será feita de forma mais segura e controlada.

A ideia é permitir login apenas para usuários que estejam no grupo:

```text
GRP_HOMELAB_ADMINS
```

Com isso, usuários como abaixo poderão autenticar, desde que estejam nesse grupo:

```text
jmsalles
adm-jeferson
```

Essa abordagem evita que uma conta de serviço, por exemplo, consiga logar no Gitea apenas por existir no AD.

## 4. Observação sobre o usuário local admin do Gitea

Antes de ativar LDAP, mantenha um usuário local administrador no Gitea funcionando.

Exemplo:

```text
jmsalles
```

ou:

```text
gitea-admin
```

Não remova nem bloqueie o administrador local até validar completamente o login LDAP.

Motivo:

```text
Se a configuração LDAP ficar incorreta, você ainda terá uma conta local para acessar o painel administrativo do Gitea.
```

## 5. Observação sobre e-mail no AD

O Gitea precisa de um atributo para preencher o e-mail do usuário.

Em muitos ambientes Active Directory, o atributo usado é:

```text
mail
```

Porém, no seu homelab, como os usuários foram criados inicialmente via PowerShell, o atributo `mail` pode não estar preenchido.

Por isso, neste guia será usado inicialmente:

```text
userPrincipalName
```

Exemplo:

```text
jmsalles@jmsalles.homelab.br
```

Se futuramente você preencher o atributo `mail` no AD, poderá alterar o campo do Gitea para:

```text
mail
```

## 6. Validar conectividade entre Gitea e AD

No servidor Docker HML, teste a porta LDAP:

```bash
nc -vz 192.168.31.24 389
```

Se o comando `nc` não existir:

```bash
sudo dnf install -y nmap-ncat
```

Teste também por nome:

```bash
nc -vz winserver.jmsalles.homelab.br 389
```

Valide DNS:

```bash
nslookup winserver.jmsalles.homelab.br
```

Se o nome não resolver corretamente, use o IP na configuração do Gitea:

```text
192.168.31.24
```

## 7. Testar consulta LDAP antes de configurar o Gitea

No servidor Docker HML ou em outro Linux, instale o cliente LDAP se necessário:

```bash
sudo dnf install -y openldap-clients
```

Teste a conta de bind:

```bash
ldapwhoami -x \
  -H ldap://192.168.31.24:389 \
  -D "svc-ldap-bind@jmsalles.homelab.br" \
  -W
```

Resultado esperado:

```text
u:JMSALLES\svc-ldap-bind
```

Teste a busca do usuário `jmsalles`:

```bash
ldapsearch -x \
  -H ldap://192.168.31.24:389 \
  -D "svc-ldap-bind@jmsalles.homelab.br" \
  -W \
  -b "OU=Homelab,DC=jmsalles,DC=homelab,DC=br" \
  "(&(objectClass=user)(sAMAccountName=jmsalles))" \
  distinguishedName sAMAccountName userPrincipalName memberOf
```

Teste o usuário `adm-jeferson`:

```bash
ldapsearch -x \
  -H ldap://192.168.31.24:389 \
  -D "svc-ldap-bind@jmsalles.homelab.br" \
  -W \
  -b "OU=Homelab,DC=jmsalles,DC=homelab,DC=br" \
  "(&(objectClass=user)(sAMAccountName=adm-jeferson))" \
  distinguishedName sAMAccountName userPrincipalName memberOf
```

O retorno deve mostrar o usuário e, preferencialmente, o grupo:

```text
CN=GRP_HOMELAB_ADMINS,OU=Groups,OU=Homelab,DC=jmsalles,DC=homelab,DC=br
```

## 9. Preencher os campos principais

Na tela `Adicionar Fonte de Autenticação`, preencha conforme abaixo.

## 9.1 Tipo de Autenticação

Campo:

```text
Tipo de Autenticação
```

Valor:

```text
LDAP (via BindDN)
```

## 9.2 Nome da Autenticação

Campo:

```text
Nome da Autenticação
```

Valor recomendado:

```text
AD JMSALLES
```

## 9.3 Protocolo de Segurança

Campo:

```text
Protocolo de Segurança
```

Valor inicial:

```text
Unencrypted
```

Motivo:

```text
Neste momento o AD está sendo usado via LDAP simples na porta 389.
Futuramente é recomendado evoluir para LDAPS na porta 636.
```

## 9.4 Servidor

Campo:

```text
Servidor
```

Valor recomendado, se DNS estiver funcionando:

```text
winserver.jmsalles.homelab.br
```

Valor alternativo, se o container do Gitea não resolver o nome:

```text
192.168.31.24
```

Para o primeiro teste, se quiser reduzir variáveis, use:

```text
192.168.31.24
```

## 9.5 Porta

Campo:

```text
Porta
```

Valor:

```text
389
```

## 9.6 Vincular DN

Campo:

```text
Vincular DN
```

Valor:

```text
CN=svc-ldap-bind,OU=Service Accounts,OU=Homelab,DC=jmsalles,DC=homelab,DC=br
```

## 9.7 Vincular Senha

Campo:

```text
Vincular Senha
```

Valor:

```text
<senha_do_usuario_svc-ldap-bind>
```

## 9.8 Base de Pesquisa de Usuário

Campo:

```text
Base de Pesquisa de Usuário
```

Valor:

```text
OU=Homelab,DC=jmsalles,DC=homelab,DC=br
```

## 9.9 Filtro de Usuário

Campo:

```text
Filtro de Usuário
```

Valor recomendado para este momento:

```text
(&(objectClass=user)(sAMAccountName=%[1]s)(memberOf=CN=GRP_HOMELAB_ADMINS,OU=Groups,OU=Homelab,DC=jmsalles,DC=homelab,DC=br))
```

Esse filtro faz três coisas:

```text
1. Busca apenas objetos do tipo usuário
2. Procura pelo login digitado no Gitea usando sAMAccountName
3. Permite apenas usuários membros do grupo GRP_HOMELAB_ADMINS
```

Exemplo:

```text
Login digitado: jmsalles
Busca LDAP: sAMAccountName=jmsalles
```

## 9.10 Filtro de Administrador

Campo:

```text
Filtro de Administrador
```

Valor:

```text
(memberOf=CN=GRP_HOMELAB_ADMINS,OU=Groups,OU=Homelab,DC=jmsalles,DC=homelab,DC=br)
```

Esse filtro faz com que usuários desse grupo sejam tratados como administradores no Gitea.

Atenção:

```text
Use esse filtro apenas se você realmente quiser que os membros do grupo GRP_HOMELAB_ADMINS sejam administradores do Gitea.
```

## 9.11 Filtro de Restrição

Campo:

```text
Filtro de Restrição
```

Valor:

```text
Deixar em branco
```

Neste momento, não vamos usar restrição adicional.

## 19. Opção futura com grupo próprio do Gitea

O ideal futuramente é criar um grupo específico para usuários do Gitea, por exemplo:

```text
GRP_GITEA_USERS
```

E outro para administradores:

```text
GRP_GITEA_ADMINS
```

Nesse modelo futuro, o filtro de usuário ficaria assim:

```text
(&(objectClass=user)(sAMAccountName=%[1]s)(memberOf=CN=GRP_GITEA_USERS,OU=Groups,OU=Homelab,DC=jmsalles,DC=homelab,DC=br))
```

E o filtro de administrador ficaria assim:

```text
(memberOf=CN=GRP_GITEA_ADMINS,OU=Groups,OU=Homelab,DC=jmsalles,DC=homelab,DC=br)
```

Neste momento, como você já tem o grupo `GRP_HOMELAB_ADMINS`, vamos usá-lo para simplificar.

## 20. Opção futura com LDAPS

Neste momento, a configuração usa LDAP simples:

```text
ldap://192.168.31.24:389
```

Futuramente, o ideal é evoluir para LDAPS:

```text
ldaps://winserver.jmsalles.homelab.br:636
```

Para isso será necessário:

```text
Configurar certificado no Domain Controller
Garantir que o certificado tenha o FQDN correto
Instalar a CA raiz no container/host do Gitea
Alterar Protocolo de Segurança para LDAPS ou TLS conforme a tela do Gitea
Alterar a porta para 636
Validar a cadeia de confiança
```

Teste futuro de LDAPS:

```bash
openssl s_client -connect winserver.jmsalles.homelab.br:636 -showcerts
```

## 21. Troubleshooting

## 21.1 Login LDAP falha

Valide primeiro a senha do usuário diretamente no AD:

```bash
ldapwhoami -x \
  -H ldap://192.168.31.24:389 \
  -D "jmsalles@jmsalles.homelab.br" \
  -W
```

Teste também:

```bash
ldapwhoami -x \
  -H ldap://192.168.31.24:389 \
  -D "adm-jeferson@jmsalles.homelab.br" \
  -W
```

Se falhar aqui, o problema não é no Gitea.

Verifique:

```text
Senha incorreta
Usuário bloqueado
Usuário desabilitado
Senha expirada
Necessidade de trocar senha no próximo logon
```

## 21.2 Bind falhando

Teste a conta de bind:

```bash
ldapwhoami -x \
  -H ldap://192.168.31.24:389 \
  -D "svc-ldap-bind@jmsalles.homelab.br" \
  -W
```

Se falhar, revise:

```text
Senha do svc-ldap-bind
DN do svc-ldap-bind
Conta habilitada
Porta 389 acessível
```

## 21.3 Usuário não é encontrado

Teste a busca:

```bash
ldapsearch -x \
  -H ldap://192.168.31.24:389 \
  -D "svc-ldap-bind@jmsalles.homelab.br" \
  -W \
  -b "OU=Homelab,DC=jmsalles,DC=homelab,DC=br" \
  "(&(objectClass=user)(sAMAccountName=jmsalles))" \
  distinguishedName sAMAccountName memberOf
```

Se não retornar, valide:

```text
Base de Pesquisa de Usuário
Filtro de Usuário
Localização do usuário no AD
sAMAccountName correto
```

## 21.4 Usuário existe, mas não passa no filtro

Se o filtro usado for este:

```text
(&(objectClass=user)(sAMAccountName=%[1]s)(memberOf=CN=GRP_HOMELAB_ADMINS,OU=Groups,OU=Homelab,DC=jmsalles,DC=homelab,DC=br))
```

o usuário precisa estar no grupo:

```text
GRP_HOMELAB_ADMINS
```

Valide:

```bash
ldapsearch -x \
  -H ldap://192.168.31.24:389 \
  -D "svc-ldap-bind@jmsalles.homelab.br" \
  -W \
  -b "OU=Homelab,DC=jmsalles,DC=homelab,DC=br" \
  "(sAMAccountName=jmsalles)" \
  memberOf
```

Se o grupo não aparecer em `memberOf`, o login será bloqueado pelo filtro.

## 21.5 Usuário loga, mas não vira admin

Valide o campo:

```text
Filtro de Administrador
```

Ele deve estar assim:

```text
(memberOf=CN=GRP_HOMELAB_ADMINS,OU=Groups,OU=Homelab,DC=jmsalles,DC=homelab,DC=br)
```

Valide também se o usuário é membro desse grupo.

## 21.6 Erro por e-mail vazio

Se o Gitea reclamar de e-mail ou não criar o usuário corretamente, altere o campo:

```text
Atributo do E-mail
```

De:

```text
mail
```

Para:

```text
userPrincipalName
```

Motivo:

```text
No AD do homelab, o atributo mail pode não estar preenchido.
O userPrincipalName normalmente já possui formato de e-mail.
```

## 21.7 Erro de DNS no container do Gitea

Se o Gitea não resolver:

```text
winserver.jmsalles.homelab.br
```

use no campo `Servidor`:

```text
192.168.31.24
```

Depois, futuramente, ajuste DNS do container/host para usar o DNS do AD.

## 21.8 Fonte LDAP criada incorretamente

Se precisar corrigir, acesse:

```text
Configurações de Administrador
Identidade e Acesso
Fontes de Autenticação
AD JMSALLES
```

Ajuste os campos e salve.

## 21.9 Usuário local com mesmo nome

Se já existir um usuário local `jmsalles` no Gitea, pode haver conflito dependendo de como ele foi criado.

Cenário comum:

```text
Usuário local: jmsalles
Usuário LDAP: jmsalles
```

Se esse usuário já é seu admin local, valide com cuidado antes de alterar autenticação.

Recomendação:

```text
Mantenha pelo menos uma conta admin local separada, como gitea-admin, para contingência.
```
