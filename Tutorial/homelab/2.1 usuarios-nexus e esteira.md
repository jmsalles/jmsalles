# Guia de criação dos usuários de serviço do homelab JMSalles

## 1. Objetivo

Este documento descreve como criar os usuários de serviço necessários para a esteira do homelab JMSalles.

Usuários contemplados:

```text
svc-gitea-runner
svc-k8s-pull
svc-docker-deploy
```

Uso de cada usuário:

| Usuário             | Local                 | Finalidade                                                                            |
| ------------------- | --------------------- | ------------------------------------------------------------------------------------- |
| `svc-gitea-runner`  | Nexus                 | Usado pela esteira do Gitea/act_runner para fazer login e push de imagens no registry |
| `svc-k8s-pull`      | Nexus                 | Usado pelo Kubernetes para fazer pull de imagens privadas                             |
| `svc-docker-deploy` | Linux no `docker-hml` | Usado pela esteira para executar deploy em Docker/HML                                 |

Neste modelo, o usuário `svc-docker-deploy` terá:

```text
Login via chave SSH
Login via usuário e senha
Permissão direta ao Docker via grupo docker
```

Observação importante:

```text
Usuário no grupo docker possui privilégio elevado no host.
Na prática, acesso ao grupo docker pode equivaler a acesso administrativo.
Para o homelab e esteira controlada, seguiremos com essa abordagem simplificada.
```

---

## 4. Acessar o Nexus

## 5. Criar role para push no Docker hosted

No Nexus, acesse:

```text
Settings
Security
Roles
Create role
Nexus role
```

Preencha:

| Campo       | Valor                                                     |
| ----------- | --------------------------------------------------------- |
| Role ID     | `role-docker-hosted-push`                                 |
| Role name   | `Role Docker Hosted Push`                                 |
| Description | `Permissao para push e pull no repositorio docker-hosted` |

Em `Privileges`, procure por:

```text
nx-repository-view-docker-docker-hosted
```

Adicione:

```text
nx-repository-view-docker-docker-hosted-browse
nx-repository-view-docker-docker-hosted-read
nx-repository-view-docker-docker-hosted-add
nx-repository-view-docker-docker-hosted-edit
```

Não adicione `delete` inicialmente.

Motivo:

```text
O usuario da esteira deve conseguir publicar e atualizar imagens, mas nao precisa apagar artefatos.
```

Salve a role.

---

## 6. Criar role para pull no Docker hosted

No Nexus, acesse:

```text
Settings
Security
Roles
Create role
Nexus role
```

Preencha:

| Campo       | Valor                                                    |
| ----------- | -------------------------------------------------------- |
| Role ID     | `role-docker-hosted-pull`                                |
| Role name   | `Role Docker Hosted Pull`                                |
| Description | `Permissao somente leitura no repositorio docker-hosted` |

Em `Privileges`, adicione:

```text
nx-repository-view-docker-docker-hosted-browse
nx-repository-view-docker-docker-hosted-read
```

Salve a role.

Essa role será usada pelo usuário:

```text

```

# Parte 4: Criar usuário Linux `svc-docker-deploy`

## 14. Criar usuário no servidor Docker HML

Acesse o servidor:

```bash
ssh jmsalles@docker-hml.jmsalles.homelab.br
```

Crie o usuário:

```bash
sudo useradd -m -s /bin/bash svc-docker-deploy
```

Defina senha para permitir login com usuário e senha:

```bash
sudo passwd svc-docker-deploy
```

Informe uma senha forte.

Valide o usuário:

```bash
id svc-docker-deploy
```

Valide o shell:

```bash
getent passwd svc-docker-deploy
```

Resultado esperado:

```text
svc-docker-deploy:x:<UID>:<GID>::/home/svc-docker-deploy:/bin/bash
```

---

## 15. Criar chave pública e privada para `svc-docker-deploy`

A chave deve ser criada na máquina que vai executar a esteira, por exemplo:

```text
Servidor do Gitea act_runner
Notebook administrativo
Servidor de CI/CD
```

A chave privada ficará na esteira ou futuramente no Vault.

A chave pública será adicionada no servidor Docker HML, no usuário:

```text
svc-docker-deploy
```

---

### 15.1 Criar diretório local para as chaves

Na máquina de origem, execute:

```bash
mkdir -p ~/.ssh/jmsalles-homelab
```

```bash
chmod 700 ~/.ssh
```

```bash
chmod 700 ~/.ssh/jmsalles-homelab
```

---

### 15.2 Gerar a chave SSH

Crie a chave:

```bash
ssh-keygen -t ed25519 -f ~/.ssh/jmsalles-homelab/svc-docker-deploy -C "svc-docker-deploy@docker-hml"
```

Quando pedir senha da chave, para uso automatizado pela esteira pode deixar sem senha.

Arquivos criados:

```text
~/.ssh/jmsalles-homelab/svc-docker-deploy
~/.ssh/jmsalles-homelab/svc-docker-deploy.pub
```

Ajuste as permissões:

```bash
chmod 600 ~/.ssh/jmsalles-homelab/svc-docker-deploy
```

```bash
chmod 644 ~/.ssh/jmsalles-homelab/svc-docker-deploy.pub
```

Valide:

```bash
ls -l ~/.ssh/jmsalles-homelab/
```

---

### 15.3 Exibir a chave pública

Exiba a chave pública:

```bash
cat ~/.ssh/jmsalles-homelab/svc-docker-deploy.pub
```

Copie o conteúdo completo da chave pública.

O formato será parecido com:

```text
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIExemploDeChavePublica svc-docker-deploy@docker-hml
```

---

### 15.4 Criar o diretório `.ssh` no usuário `svc-docker-deploy`

No servidor Docker HML, execute:

```bash
sudo mkdir -p /home/svc-docker-deploy/.ssh
```

Crie ou edite o arquivo `authorized_keys`:

Ajuste o dono:

```bash

```

Ajuste as permissões:

```bash
sudo chmod 700 /home/svc-docker-deploy/.ssh
```

```bash
sudo chmod 600 /home/svc-docker-deploy/.ssh/authorized_keys
```

Valide:

```bash
sudo ls -lah /home/svc-docker-deploy/.ssh
```

## 16. Adicionar `svc-docker-deploy` ao grupo `docker`

Neste modelo, o usuário `svc-docker-deploy` terá permissão direta para executar comandos Docker.

Observação importante:

```text
Usuário no grupo docker possui privilégio elevado no host.
Na prática, acesso ao grupo docker pode equivaler a acesso administrativo.
Para o homelab e esteira controlada, seguiremos com essa abordagem simplificada.
```

Adicione o usuário ao grupo `docker`.

```bash
sudo usermod -aG docker svc-docker-deploy
```

Valide:

```bash
id svc-docker-deploy
```

Resultado esperado:

```text
svc-docker-deploy deve aparecer como membro do grupo docker.
```

Exemplo:

```text
groups=...,docker
```

## 17. Permitir login SSH com usuário e senha para `svc-docker-deploy`

Neste item, será permitido que o usuário `svc-docker-deploy` acesse o servidor também com senha.

Valide primeiro se o usuário tem senha ativa:

```bash
sudo passwd -S svc-docker-deploy
```

Se aparecer `L`, o usuário está bloqueado para senha.

Desbloqueie, se necessário:

```bash
sudo passwd -u svc-docker-deploy
```

Redefina a senha, se necessário:

```bash
sudo passwd svc-docker-deploy
```

---

### 17.1 Criar configuração SSH específica para o usuário

Crie um arquivo dedicado para o usuário:

```bash
sudo vim /etc/ssh/sshd_config.d/99-svc-docker-deploy.conf
```

Conteúdo:

```text
Match User svc-docker-deploy
    PasswordAuthentication yes
    PubkeyAuthentication yes
    KbdInteractiveAuthentication yes
```

Observação:

```text
Essa configuração permite senha e chave SSH apenas para o usuário svc-docker-deploy.
```

---

### 17.2 Validar configuração do SSH

Valide a configuração:

```bash
sudo sshd -t
```

Se não retornar erro, recarregue o serviço SSH.

Em Ubuntu/Debian, normalmente:

```bash
sudo systemctl reload ssh
```

Em Rocky/RHEL, normalmente:

```bash
sudo systemctl reload sshd
```

Se quiser usar um comando compatível com ambos:

```bash
sudo systemctl reload ssh || sudo systemctl reload sshd
```

# Parte 6: Como a esteira usará os usuários

## 21. Uso do `svc-gitea-runner` na pipeline

O `svc-gitea-runner` será usado para login e push no Nexus.

Exemplo conceitual da pipeline:

```bash
docker login registry.jmsalles.homelab.br -u svc-gitea-runner -p "$NEXUS_PASSWORD"
```

```bash
docker build -t registry.jmsalles.homelab.br/lab/app-teste:latest .
```

```bash
docker push registry.jmsalles.homelab.br/lab/app-teste:latest
```

A senha do usuário deve ficar futuramente no Vault.

---

## 22. Uso do `svc-k8s-pull` no Kubernetes

O `svc-k8s-pull` será usado no `imagePullSecret`.

Exemplo:

```bash
kubectl create secret docker-registry nexus-registry \
  --docker-server=registry.jmsalles.homelab.br \
  --docker-username=svc-k8s-pull \
  --docker-password='<SENHA_DO_USUARIO>' \
  --docker-email='jefersonmattossalles@gmail.com' \
  -n hml
```

No deployment Kubernetes:

```yaml
imagePullSecrets:
  - name: nexus-registry
```

Imagem:

```yaml
image: registry.jmsalles.homelab.br/lab/app-teste:latest
```

---

## 23. Uso do `svc-docker-deploy` na esteira Docker

A esteira poderá acessar o host Docker HML via SSH.

Exemplo usando chave SSH:

```bash
ssh -i ~/.ssh/jmsalles-homelab/svc-docker-deploy \
  svc-docker-deploy@docker-hml.jmsalles.homelab.br \
  "cd /home/jmsalles/minha-app && docker compose pull && docker compose up -d"
```

Exemplo usando usuário e senha:

```bash
ssh svc-docker-deploy@docker-hml.jmsalles.homelab.br \
  "cd /home/jmsalles/minha-app && docker compose pull && docker compose up -d"
```

Restart controlado:

```bash
ssh svc-docker-deploy@docker-hml.jmsalles.homelab.br \
  "cd /home/jmsalles/minha-app && docker compose restart"
```

Ver logs:

```bash
ssh svc-docker-deploy@docker-hml.jmsalles.homelab.br \
  "cd /home/jmsalles/minha-app && docker compose logs --tail=100"
```

Observação:

```text
Para o Nexus, evite deploy automático sem janela, porque ele é componente central do registry.
```
